INTRODUÇÃO

Este manual contém todas as práticas de desenvolvimento da equipe Isabela Flores.

Pressupostos

Não crie seus códigos para que somente Deus entenda!

Evite Codificações: não crie código. Somos programadores, não criptólogos. Ao desenvolver uma aplicação, um sistema, o que for, é necessário fazê-lo de forma que seja possível entender a qualquer tempo, seja no decorrer do desenvolvimento, seja daqui um bom tempo.

ESCREVA PARA PROGRAMADORES

Ao escrever o código é necessário ter em mente, que outro programador terá que manter o código escrito por você. Você não é imortal, um dia você pode estar doente, ausente, ou até mesmo um dia sairá da linha de frente e se dedicará a parte administrativa da empresa, nesse caso terá que deixar seu código para um outro programador dar continuidade nas futuras atualizações, e dedicar seu tempo nos negócios da empresa.

É nessas horas que um código bem escrito, feito para pessoas e não para máquinas tem suas vantagens. O programador vai ter a sua frente um código enxuto e de fácil manutenção.

Ambiente de Produção - Regras Importantes

  • Nunca deixar funções do tipo var_dump(), error_log(), etc no código.
  • Nunca faça mais requisições ajax que necessário. Tente trazer todas as informações necessárias em apenas uma requisição.
  • Nunca retorne mais do que o necessário nas requisições ajax. Se precisar do nome do cliente para mostrar na tela, não é necessário retornar com ajax as tags, o nome, o style, etc para ser mostrado na tela, já deixe toda estrutura pronta no html e apenas retorne o nome do cliente puro e simplesmente com ajax.
  • Use o máximo de cache possível. Apenas não cacheie informações que podem variar por usuário.
  • Cuidado com os códigos e comentários feitos em HTML e JavaScript eles aparecem inteiros no navegador.
  • Usa apenas recursos computacionais necessários (memória, processador)! Não inicialize mais variáveis do que vai usar, não crie código complexos com inúmeras linhas, MENOS É MAIS!
  • Código inútil: todo código que não está sendo utilizado deve ser descartado. Não há necessidade de se manter algo inútil.
  • NUNCA use loop dentro de loop, isso é ineficiência de lógica! Repense a lógica do seu código!

Abreviações nos nomes

Evite usar, se necessário use, mas somente use abreviações se ela for de conhecimento geral e se seguir o padrão do resto do sistema. Por ex.: se em algum lugar no sistema já se abrevia pagamento como pgto, não vá tentar ser o diferente e abreviar como pgmto ou pmto, o mínimo que você vai conseguir é não ser entendido.

Abreviações padrões:

  • pagamento: pgto
  • data: dt
  • pedido: ped
  • produto: prod
  • valor: vlr
  • hora: hr
  • floricultura: floric
  • número: num
  • nota fiscal: nfe
  • negociador: neg
  • follow up: fup
  • confirmação de entrega: ce
  • estorno: est

Variáveis

  • Só criar variáveis no escopo que ela será usada;
  • Sempre que não for mais usar uma variável, limpe o endereço de memória dela com o comando: unset($variavel);
  • Nome da variável: Usar o nome que identifica o que exatamente essa variável armazena, seja sucinto, você deve ser capaz de saber o que ela contém apenas com ele, "valor", "igual", "dados" não são nomes válidos para nenhum caso. você deve ser capaz de saber o que ela contém;
  • Use Nomes Pronunciáveis: Como function hndlimghght, mas se fosse um function manipularTamanhoImagem, milhares de vezes mais fácil de memorizar, dizer para o colega do lado e de procurar no meio de centenas ou milhares de linhas de código.
    Ex.: Preciso armazenar numa variável a data e hora de hoje para usar na minha função. Nome da variável: $data_hora_atual;
  • Não inicie nomes de variáveis com o_, obj_, m_, etc. Uma variável não precisa de prefixos indicando o estado da variável;
  • Use apenas palavras em português;

Funções e métodos

  • Devem ter um nome que representa o que aquele código faz como verbo(nunca com nome do que ela retorna, ou do que ela recebe, e sim qual a função daquele código).
  • Será no singular ou plural dependendo do retorno. (Se retornar um array de clientes, a função terá o nome no plural, se salvar apenas um cliente, a função será no singular)
  • Devem começar com letra minúscula e da segunda palavra adiante a primeira letra da palavra em maiúsculo.
  • SEMPRE usar os comentários de início de função que explicam mais sobre a função. Ver mais em Comentários.
  • Faça apenas uma coisa: os métodos devem fazer uma coisa e devem fazê-la bem. O motivo de criarmos uma função é para decompor um conceito maior em uma série de passos no próximo nível de abstração.

Ex.: 

                      function salvarCliente(){
                        //..    
                      }

Ex. de nomes com começos mais comuns (enviar,salvar,consultar,calcular,editar): 

                      function enviarEmailNovoPedido(){}
                      function salvarPedido(){}
                      function consultarClientes(){}
                      function calcularValorCompra(){}
                      function editarCliente(){}

  • Funções começadas em get e set só podem ser usadas em classes de objetos para pegar ou setar um atributo dentro dele.
  • As classes devem ser pequenas: assim como métodos ou funções, a primeira regra para classes é que devem ser pequenas. A segunda é que devem ser menores ainda. A classe deve possuir poucas responsabilidades. O nome da classe deve descrever suas responsabilidades.

Use sempre DRY

Don’t Repeat Yourself(ou simplesmente, não se repita). Como o nome indica, é garantir que você não escreva código redundante.

Ex.: Neste exemplo se precisar alterar alguma variável de conexão é simples com o DRY já que as variáveis estarão no começo do código. 

                      //CÓDIGO SEM DRY:
                      $mysql = mysql_connect('localhost', 'reinhold', 'secret_hash');
                      mysql_select_db('wordpress') or die("cannot select DB");
                    
                      //Agora, com a abordagem DRY:
                      $bd_host = 'localhost';
                      $bd_usuario = 'reinhold';
                      $bd_senha = 'secret_hash';
                      $bd_banco_dados = 'wordpress';
                         
                      $mysql = mysql_connect($bd_host, $bd_usuario, $bd_senha);
                      mysql_select_db($bd_banco_dados);

Mensagens de retorno de erro, sucesso e alerta

  • Mensagem de Erro:

É importante essas mensagens informarem ao usuário de forma simples e sucinta 3 coisas: SEM FIRULAS
1- Qual erro ocorreu sem muitos detalhamentos, escreva uma mensagem para sua vó.
2- O que deve ser feito para não dar erro. Explique de forma MUITO simples o que fazer para sua vovó.
3- De forma complexa e rápida mostre algo para que um programador possa identificar o erro, caso seja algo sistêmico.

Ex.: Não foi possível salvar o pedido. Salve o endereço e tente novamente. (address not found - Erro 122)

  • Mensagem de Sucesso:

Sempre seja simples e direto. SEM FIRULAS.
Ex.: Cliente salvo com sucesso. Ex.: Preço alterado com sucesso.

  • Mensagem de Alerta:

Sempre seja simples e direto. SEM FIRULAS.

Normalização

  • Não crie base de dados com milhares de tabelas se relacionando, prefira incluir mais colunas em uma mesma tabela. Isso reduz o uso de JOINS nas consultas;

Ex.:

Ao invés de criar:

tables: (cliente,telefone_cliente,endereco_cliente,email_cliente)

Crie:

table: (cliente) e colunas(id_cliente,nome,telefone,endereco,email)

Databases

  • Sempre usar o prefixo db_ e o nome da database. Sempre com letra minúscula.

Ex.: db_joanaflores, db_mariaflores, db_pico

Tabelas

  • Usar o prefixo adequado entre: e todo nome sempre com letra minúscula

tbd_ - tabela de dados - Ex.: tabela de produtos, clientes, endereços
tbi_ - tabela de informações - Ex.: tabela de status
tbr_ - tabela relacional - Ex.: tabela que relaciona uma tabela de cliente e uma tabela de endereçõs (n pra n)
tbl_ - tabela de log/histórico de alterações.

  • O nome da tabela deve ser curto e simples identificando o que contém aquela tabela

Ex.:

tbd_pedido
tbd_produto
tbi_status
tbd_floricultura
tbd_cidade
tbr_floricultura_cidade

Nomenclaturas de tabela e colunas

  • Nunca usar letra maiúscula;
  • Não usar espaço em branco;
  • Não usar hífen, acentos e caracteres especiais;
  • Não usar verbos;
  • Não utilizar palavras no plural;
  • Não usar números;
  • Separe as palavras com underline;
  • Crie nomes sucintos e objetivos, não pode ter várias interpretações;
  • ID - São chamados de id e não código;
  • Nomes de Colunas Primary Keys: id
  • Nomes de Colunas Foreign Keys: nome_tabela_nome_coluna

Ex. de coluna Foreign Key:

tbd_cliente_id
tbd_cidade_nome
tbd_floricultura_telefone

Ex. de tabela:

tbd_cliente
tbd_cidade

Ex. de colunas:

nome_cliente
email_cliente
id

Tipos de Colunas de Tabelas

  • Data - Tipo deve ser sempre date;
  • Data e hora - Tipo deve ser sempre datetime;
  • Booleano - Tipo deve ser sempre boolean (Se não houver esse campo, use tinyint(1));
  • Palavras com tamanho de caracteres fixo - Tipo deve ser sempre char;
    Ex.:
    Coluna CEP - Tipo char(8)
    Coluna CPF - Tipo char(11)
  • Palavras com tamanho de caracteres váriavel porém, existe um limite - Tipo deve ser sempre varchar;
    Ex.:
    Coluna Nome do Cliente - Tipo varchar(50)
    Coluna Nome da Cidade - Tipo varchar(100)
  • Palavras com tamanho de caracteres váriavel e sem um limite - Tipo deve ser sempre text;
    Ex.:
    Coluna Mensagem do Cartão de Mensagem - Tipo text
    Coluna Observação - Tipo text
  • Só use o campo text se for extremamente necessário, sempre opte pelo varchar com um limite adequado a necessidade.
  • Use sempre o tamanho correto e o tipo correto sem prejudicar o formulário e sem sobrecarregar o banco de dados;
  • Tipos de dados pequenos são mais rápidos.
    Não é necessário criarmos um campo em uma tabela com o tipo de dado INT se esse valor em sua aplicação vai chegar até no máximo 1000. Tente utilizar MEDIUMINT, SMALLINT ou mesmo TINYINT, o mesmo caso se aplica para o CHAR, as consultas ficam mais velozes em campos com VARCHAR(10) do que CHAR(10), pois, registros com VARCHAR irá ficar dessa maneira em sua tabela: “0123”, enquanto o CHAR irá ocupar todos os espaços: “0123“.

Consultas - Select

  • Use várias consultas simples isentas ou com o mínimo de JOIN ao invés de uma consulta complexa com inúmeros JOINS.
  • Sempre traga apenas as colunas necessárias, não traga mais colunas do que o necessário e muito menos faça consultas com asterisco (select * from);
  • Em WHERE e ON use sempre colunas do tipo int, só use outro tipo se não houver outra opção.
    Ex.: 
                                    select id from tbd_cliente where id_cliente = 5;
  • Fazer consultas dentro de laços de repetição pode ser extremamente danoso para o servidor, evite fortemente essa prática;
  • O comando select do MySQL oferece inúmeras opção para se fazer a mesma consulta, SEMPRE opte pela que sobrecarrega menos a memória do servidor de banco.
  • Opte SEMPRE por fazer manipulações, formatações e condições na linguagem de programação (PHP/Javascript/etc), ao invés de usar funções do MySQL para esse objetivo.
  • O servidor de banco de dados deve ser poupado! Deixe o servidor com PHP trabalhar mais que o MySQL!
  • Sempre que você tiver certeza que um registro retornará apenas um resultado inclua o LIMIT 1, isso aumenta o desempenho de sua consulta, pois, desta forma a tabela não será toda processada e sim processada até ser encontrado o primeiro resultado.
  • De maneira nenhuma use: ORDER BY RAND();
  • Monitore o arquivo /var/log/mysql/mysql-slow.log para descobrir possíveis consultas pesadas para o banco e MELHORE;
